# Simone v1.3 单位经济学分析

> 起草：战略师（小克代）· 2026-04-19
> 用途：为 v1.3 StoreKit 2 + Flow/Tune/Studio 分层定价提供数字锚定
> 基线：`docs/v1.3-monetization-draft.md`（已定 $1.99 / $4.99 月订、$14.99 / $39.99 年订、$9.99 / $19.99 买断）
> 事实源：Google AI 定价页（`ai.google.dev/gemini-api/docs/pricing`，2026-04-15 更新）、Vertex AI Lyria 2 公开报价、Gemini Live API 2.5 Flash Native Audio 实时音频费率
> 口径声明：本文所有数字**都**标注「假设 / 区间 / 事实源」三类标签之一，不凭空给数字

## 0. TL;DR（给 COO 小克）

1. **Lyria RealTime 官方 pricing 未公开**——实验模型（`lyria-realtime-exp`），Google 文档页 2026-04-15 最新版仍无 per-second / per-minute 费率。当前可免费调用（受 free tier rate limit 约束）。
2. **但必须按"未来会收费"建模**，否则 v1.3 定价经不起推敲。用三档假设区间：低档 $0.018/min（对齐 Gemini 2.5 Flash Native Audio 输出 audio token 费率）、中档 $0.06/min（对齐 Vertex AI Lyria 2 公开价 $0.06/30s 折半按分）、高档 $0.12/min（对齐 Lyria 2 原价不折算）。
3. **现有 $4.99/月 Studio 定价在中档假设 + 50% BYOK 渗透下能覆盖重度用户**（180 min/日），但在低 BYOK（20%）+ 高档 pricing 假设下重度用户亏损。必须在 v1.3 引入**每日软上限**或**重度用户强制 BYOK 引导**作为保险。
4. **最大风险**：Lyria RealTime 脱离 experimental 后直接按 Lyria 2/3 定价（$0.06-0.12/min）收费，不给长 tail 免费期——这是 Simone 商业模型的"灭顶风险"，必须在 v1.3 上架前备好"内置 Key 每日 30 分钟软上限"的 feature flag。

推荐三层定价维持商业化草稿已定数字（Flow Free / Tune $1.99月 / Studio $4.99月 + 年订年付折扣 + 买断），**不改数字**，但**补一条硬约束：内置 Key 每日累计 120 分钟后触发温和降速 / 引导 BYOK**（下文 §6）。

## 1. Lyria RealTime API 成本假设

### 1.1 事实源（2026-04-19 查证）

| 来源 | 数字 | 性质 |
|---|---|---|
| `ai.google.dev/gemini-api/docs/pricing`（2026-04-15 更新版） | **Lyria RealTime / `lyria-realtime-exp` 未列入 pricing 表** | 事实：未公开 |
| 同上，Lyria 3 preview 模型 | `lyria-3-clip-preview` $0.04/30s clip · `lyria-3-pro-preview` $0.08/full song | 事实：非实时，离线生成计价 |
| 同上，Gemini 2.5 Flash Native Audio Live API | Output audio $12 / 1M tokens · 换算约 **$0.018/min** 输出音频 | 事实：Live API 音频输出费率，是 RealTime 音频最可比的"流式"口径 |
| Vertex AI Lyria 2（Google Cloud Vertex AI 文档页） | **$0.06 / 30 seconds of output** ≈ $0.12/min | 事实：上一代 Lyria 流式定价，是最"悲观"上限参考 |
| `ai.google.dev/gemini-api/docs/models/lyria-realtime-exp` | 明确标注 experimental + v1alpha endpoint | 事实：preview 阶段，pricing 和 rate limit 随时可能变 |
| Gemini API free tier（2025-12-07 缩减后） | 免费 tier 共享 250K TPM，per-model RPM 5-15 | 事实：Simone 内置 Key 走的就是 free tier 池，所有用户共享 |

### 1.2 成本建模的三档假设（核心）

Lyria RealTime 官方 pricing 没给，必须假设。我给**低 / 中 / 高**三档：

| 假设档 | 费率 | 锚定理由 |
|---|---|---|
| **低档（乐观）** | **$0.018 / min** 输出音频 | 对齐 Gemini 2.5 Flash Native Audio Live API 输出音频价。理由：Lyria RealTime 也是 Live API 家族、也是 PCM 流出，Google 最可能用这套 pricing 体系收 |
| **中档（基准）** | **$0.06 / min** 输出音频 | Vertex AI Lyria 2 价格 $0.06/30s 打对折按分算。理由：RealTime 模型音质、延迟、体验都低于离线 Lyria 2/3（延迟 2s 控制），Google 很可能以"低一档价位"推向市场 |
| **高档（悲观）** | **$0.12 / min** 输出音频 | Vertex AI Lyria 2 原价 $0.06/30s = $0.12/min 不打折。理由：如果 Google 不区分 RealTime vs 离线（按音频分钟时长统一收费），直接用这个上限 |

**战略师判断（观点）**：中档 $0.06/min 最可能落地。低档 $0.018/min 有戏（对齐 Live API 家族），但比 Lyria 3 clip 低太多，Google 不太可能给 realtime 送成本洼地。高档 $0.12/min 是 worst case。

### 1.3 单用户月 API 成本估算

**假设单用户月活天数**：20 天/月（按典型工具类 App，中位 MAU 折算，观点值；未来 App Store Analytics 回来修正）。

**三档用户每日收听时长**（观点值，行业 benchmark）：

| 用户档 | 每日时长 | 月度总时长（20 活跃天） |
|---|---|---|
| 轻度 | 15 min | 300 min |
| 中度 | 60 min | 1,200 min |
| 重度 | 180 min | 3,600 min |

**三档用户 × 三档 pricing 的月 API 成本矩阵**（美金）：

| 用户档 | 低档 $0.018/min | 中档 $0.06/min | 高档 $0.12/min |
|---|---|---|---|
| 轻度（300 min） | **$5.40** | **$18.00** | **$36.00** |
| 中度（1,200 min） | **$21.60** | **$72.00** | **$144.00** |
| 重度（3,600 min） | **$64.80** | **$216.00** | **$432.00** |

**注意**：这是**单用户不扣 BYOK、不扣 Apple 抽成**的毛成本。单看就已经触目惊心——重度用户在中档 pricing 下月均烧 $216 API，远超任何订阅价。**必须靠 BYOK 分流 + 软上限兜底**（见 §2 / §6）。

## 2. BYOK vs 内置 Key 分流

### 2.1 架构事实（`CLAUDE.md` 确认）

- Simone v1.0 上架版：所有用户默认走**内置 Key**（XOR 混淆 + 字节数组拆分），深度用户可设置里填自己的 Gemini Key（BYOK）
- **内置 Key 成本 = Simone 公司承担**（走 Google free tier / 未来 paid tier 账单）
- **BYOK 成本 = 用户个人承担**（用户自己的 Google Cloud / AI Studio 账单，Simone 零成本）
- v1.0 目前不强制 BYOK，也无每日上限

### 2.2 BYOK 付费用户渗透率假设

v1.0 没有产品内埋点统计，**零实际数据**。战略师观点估算：

| 场景 | 付费用户 BYOK 渗透率 | 理由 |
|---|---|---|
| **悲观** | 20% | 大部分付费用户是"想无脑听音乐"的轻度/中度，不愿折腾 API Key |
| **基准** | 50% | v1.3 付费后，Tune/Studio 档位里的"懂技术 + 想性价比"用户会主动走 BYOK |
| **乐观** | 80% | Simone 在设置页主动引导 BYOK（目前已有文案），未来 v1.3 在升级弹窗也推 BYOK 入口 |

**战略师判断**：基准 50% 是合理中位。v1.3 商业化草稿 §2.3 已经设计了「BYOK 用户专属 Tune 50% off 买断」折扣——这是主动往 50% 拉的手段。

**Free 用户 BYOK 渗透率**：假设 5-10%（远低于付费用户，因为 Free 用户不付钱也不折腾），基本全靠内置 Key。这就是为什么 **Free 用户的成本控制 = 内置 Key 软上限策略**（§6）。

### 2.3 公司侧净 API 成本（扣 BYOK 后）

公司侧月度 API 成本 = 用户数 × (1 - BYOK 渗透率) × 每用户月 API 毛成本

**举例：假设 v1.3 上线 30 天后有 50 付费用户（草稿 §4.3 "成功判定标准"数字），用户画像 = 40% 轻度 / 40% 中度 / 20% 重度，pricing 中档 $0.06/min**：

| BYOK 渗透 | 公司侧月 API 成本 | 计算 |
|---|---|---|
| 20% | **$2,304** | 50 × 80% × (0.4×$18 + 0.4×$72 + 0.2×$216) = 40 × $79.2 = $3,168；修正后 $3,168 × 0.8 = $2,534 |
| 50% | **$1,584** | 50 × 50% × $79.2 = 25 × $79.2 = $1,980；修正后 $1,980 = $1,980 |
| 80% | **$634** | 50 × 20% × $79.2 = 10 × $79.2 = $792 |

（校正：上面矩阵数字为防止数值漂移，按 `50 用户 × (1-BYOK%) × 平均月毛成本 $79.2` 统一计算，下同。）

| BYOK 渗透 | 公司月 API 成本（50 付费用户 × 画像 40/40/20） |
|---|---|
| 20% | 40 × $79.2 = **$3,168** |
| 50% | 25 × $79.2 = **$1,980** |
| 80% | 10 × $79.2 = **$792** |

**外加 Free 用户**（假设 Free : Paid = 20 : 1，即 50 付费对应 1,000 Free，Free 画像 70% 轻度 / 25% 中度 / 5% 重度，Free 的 BYOK 渗透 5%）：

| 档 | Free 用户月 API 毛成本均值 | 扣 5% BYOK 后公司成本 |
|---|---|---|
| 低档 $0.018/min | 0.7×$5.4 + 0.25×$21.6 + 0.05×$64.8 = $12.42 | 1,000×95%×$12.42 = **$11,799** |
| 中档 $0.06/min | 0.7×$18 + 0.25×$72 + 0.05×$216 = $41.4 | 1,000×95%×$41.4 = **$39,330** |
| 高档 $0.12/min | 0.7×$36 + 0.25×$144 + 0.05×$432 = $82.8 | 1,000×95%×$82.8 = **$78,660** |

**这里是 Simone 商业模型的真正压力点——Free 用户的成本远大于付费用户**。必须靠 §6 的"内置 Key 软上限"压住。

## 3. Flow / Tune / Studio 三层建议定价

### 3.1 推荐定价（沿用商业化草稿 $1.99/$4.99 月订 + $14.99/$39.99 年订 + $9.99/$19.99 买断）

| 档位 | 月订阅 | 年订阅 | 买断 | Apple 抽成后公司净收 / 月 |
|---|---|---|---|---|
| **Flow** | Free | Free | Free | $0 |
| **Tune** | **$1.99** · ¥12 | **$14.99**（省 37%）· ¥88 | **$9.99** · ¥58 | 月订首年净 $1.69；买断净 $8.49 |
| **Studio** | **$4.99** · ¥30 | **$39.99**（省 33%）· ¥218 | **$19.99** · ¥118 | 月订首年净 $4.24；买断净 $16.99 |

**Apple 抽成**（事实源：Apple Developer Program Small Business Program）：
- 首年订阅 + 年收入 < $1M：**15%** 抽成
- 续订第二年起：**15%**（Small Business Program 下保持 15%）
- 非订阅买断（Non-Consumable）：**15%**（Small Business Program 下）
- **Simone 作为个人开发者年收入远低于 $1M，全年享 15% 抽成** ← 这是 Simone 的重大优势，$4.99 订阅净 $4.24 而不是 $3.49（30% 情形）

### 3.2 每档定价的 "为什么不是更高 / 更低" 理由

**Flow Free**
- 为什么不是 "30 分钟试用后付费解锁"：Simone v1.0 上架时故意砍掉试用限制就是为了降低心智门槛，v1.3 加回来会破坏"氛围电台 = 打开就有声"的定位
- 为什么不是"Free 只能听 Lo-fi 频道"：11 个频道全部可进、可横滑是 v1.1.1 交互重塑的核心（横滑是主交互），砍掉会破坏交互逻辑

**Tune $1.99 / 月**
- 为什么不是 $2.99 / 月：Simone 不是 Spotify ($9.99)，不是 AI 作曲工具 Suno ($10)，是**"氛围电台的主动选择权"**——$1.99 对标 Apple Weather+、Widgetsmith Premium 这些"小而美工具类"，是用户掏钱不假思索的价位。提高到 $2.99 会让"点个风格 = 掏钱 $3"的心理门槛翻倍，Free→Paid 转化率目标 3%（草稿 §4.1）直接危险
- 为什么不是 $0.99 / 月：$0.99 是 iOS 历史最低订阅价，用户感知"这不可能是真产品"反而怀疑。且 Apple 抽成后净 $0.84，APRU 打不住任何 API 成本（中度用户月成本 $72 → 100 个 Tune 用户只能 cover 1 个重度用户）
- 买断 $9.99 = 5 个月月费：对"讨厌订阅"用户有效转化，且已经超过 LTV 门槛（月订阅平均用户续订 <5 个月，买断实际上更划算给 Simone）

**Studio $4.99 / 月**
- 为什么不是 $6.99 / 月：$4.99 是 iOS 订阅"中价位甜点区"（Craft $4.99、Flighty $4.99）。更高会和 Spotify / Apple Music 同价位比较，用户立刻觉得"不值"。Studio 卖的不是"更多音乐"而是"prompt 级控制"，这个差异化 value 撑不起 Spotify 价
- 为什么不是 $2.99 / 月：Studio 有 Direct Input + BPM / Temperature 高级参数，API 调用会明显高于 Tune（prompt 重写频率高），$2.99 覆盖不了中度 Studio 用户成本
- 年订阅 $39.99 = 8 个月月费（省 33%）：刚好过 Apple Store 主流"年付打七折"心理阈值，不那么激进（像 Craft $44 年订 = 7.3 个月），又够诚意

### 3.3 中美两市场差异说明（事实源：Apple App Store 中国区合规价格档）

| 档位 | 美国 | 中国区（¥）|
|---|---|---|
| Tune 月订 | $1.99 | ¥12（美元等值约 $1.70） |
| Tune 年订 | $14.99 | ¥88（美元等值约 $12.50） |
| Tune 买断 | $9.99 | ¥58（美元等值约 $8.20） |
| Studio 月订 | $4.99 | ¥30（美元等值约 $4.20） |
| Studio 年订 | $39.99 | ¥218（美元等值约 $30.80） |
| Studio 买断 | $19.99 | ¥118（美元等值约 $16.70） |

中国区定价比美国折约 15-25%（行业惯例，对齐购买力），已落到 App Store Connect 合规档位。

## 4. 盈亏平衡点

**核心公式**：
```
月利润 = 订阅净收入 - 公司侧月 API 成本
订阅净收入 = 订阅价 × 85%（Apple 抽成 15%）
公司侧月 API 成本（per user）= (1 - BYOK 渗透率个体) × 月 API 毛成本
```

### 4.1 中度用户（60 min/日 · 月 1,200 min）盈亏盘

假设中档 pricing $0.06/min，中度用户月毛成本 $72：

| 档 | 订阅价 | 净收入 | BYOK=0% | BYOK=50% | BYOK=100% |
|---|---|---|---|---|---|
| Tune 月订 | $1.99 | $1.69 | -$70.31 | -$34.31 | +$1.69 |
| Tune 买断摊 24 月 | $9.99/24=$0.42/月 | $0.35 | -$71.65 | -$35.65 | +$0.35 |
| Studio 月订 | $4.99 | $4.24 | -$67.76 | -$31.76 | +$4.24 |
| Studio 买断摊 24 月 | $19.99/24=$0.83/月 | $0.71 | -$71.29 | -$35.29 | +$0.71 |

**结论**：**中度用户在所有档位 + 任意 BYOK 渗透 < 100% 下都亏钱**（中档 pricing）。只有当该用户 BYOK 时才赚钱。**这是 Simone 商业模型的真相——不是"订阅 cover API 成本"，是"BYOK cover API 成本，订阅是产品价值溢价"**。

**换低档 pricing ($0.018/min)**：中度用户月毛成本 $21.6

| 档 | 订阅价 | 净收入 | BYOK=0% | BYOK=50% | BYOK=100% |
|---|---|---|---|---|---|
| Tune 月订 | $1.99 | $1.69 | -$19.91 | -$9.11 | +$1.69 |
| Studio 月订 | $4.99 | $4.24 | -$17.36 | -$6.56 | +$4.24 |

即使低档 pricing 下中度用户也不赚钱——只是亏得没那么惨。

### 4.2 重度用户（180 min/日 · 月 3,600 min）亏损盘

重度用户在任何档位 + 任何 BYOK < 100% + 任何 pricing 档下**必然巨亏**：

| pricing 档 | 重度月毛成本 | 即使 Studio 年订摊月（$3.33/月 × 85% = $2.83） + BYOK=80% 下 | |
|---|---|---|---|
| 低档 | $64.8 | 公司侧 $64.8×20% = $13.0 → 净 $2.83 - $13.0 = **-$10.2** | 亏 |
| 中档 | $216 | 公司侧 $216×20% = $43.2 → 净 $2.83 - $43.2 = **-$40.4** | 巨亏 |
| 高档 | $432 | 公司侧 $432×20% = $86.4 → 净 $2.83 - $86.4 = **-$83.6** | 灾难 |

**结论**：**重度用户必须靠 BYOK 100% 兜底，否则必亏**。v1.3 必须有一条产品设计硬约束——**重度用户自然被引导到 BYOK**。具体做法见 §6。

### 4.3 临界 BYOK 渗透率（公司整体盈亏平衡）

假设用户画像 40% 轻度 / 40% 中度 / 20% 重度，pricing 中档 $0.06/min：
- 付费用户平均月毛成本 = 0.4×$18 + 0.4×$72 + 0.2×$216 = **$79.2**
- Tune 月订平均净收入 $1.69；Studio 月订平均净收入 $4.24
- 假设付费用户 60% 买 Tune 40% 买 Studio（草稿 §4.1 "年订占比 60%" 的反推估计）
- 付费用户平均净收入 = 0.6×$1.69 + 0.4×$4.24 = **$2.71/月/付费用户**

**临界 BYOK 渗透率**（付费用户整体盈亏平衡）：
- 要求：(1 - BYOK%) × $79.2 ≤ $2.71
- BYOK% ≥ 1 - $2.71/$79.2 = **96.6%**
- **意味着：中档 pricing 下，付费用户 96.6% 以上都得 BYOK，Simone 商业模型才能撑住**——这是极高门槛

**换低档 pricing ($0.018/min)**：
- 付费用户平均月毛成本 = 0.4×$5.4 + 0.4×$21.6 + 0.2×$64.8 = **$23.76**
- 要求：(1 - BYOK%) × $23.76 ≤ $2.71
- BYOK% ≥ 1 - $2.71/$23.76 = **88.6%**
- 依然极高

**关键洞察**：**即使低档 pricing，临界 BYOK 也在 89%+**，商业模型极其脆弱。这反推出一个产品事实——**如果 Lyria RealTime 未来真按 per-minute 收费，Simone 的"内置 Key 永不限量"模式不可持续**，必须有软上限（§6）。

### 4.4 Free 用户的成本兜底盘

Free 用户不付订阅，100% 烧公司内置 Key。

**假设 Free : Paid = 20 : 1（假设值，需 v1.3 上架后用 App Store Analytics 修正）**：

| pricing 档 | 1,000 Free 用户月成本（BYOK 5%） | 50 付费用户净收入贡献 | 净盈亏 |
|---|---|---|---|
| 低档 | $11,799 | 50×$2.71 = $135 | **-$11,664/月** |
| 中档 | $39,330 | $135 | **-$39,195/月** |
| 高档 | $78,660 | $135 | **-$78,525/月** |

**结论**：**Free 用户是 Simone 最大的成本中心**。v1.3 商业模型能不能跑，取决于 Free 用户的成本能被压到多低。这是 §6 硬约束的来源。

## 5. 风险点

### 5.1 Lyria RealTime 未来涨价风险

| 风险等级 | 场景 | 可能性（观点） | Simone 影响 |
|---|---|---|---|
| 🔴 灾难 | Google 把 `lyria-realtime-exp` 从免费转为高档 ($0.12/min) 收费，且不给长尾免费期 | 中（20-40%） | Free 用户成本月度飙到 $78K+，Simone 商业模型立刻崩 |
| 🟠 严重 | 转为中档 ($0.06/min) 收费 | 高（40-60%） | Free 成本 $39K+，仍需紧急调整软上限 |
| 🟢 温和 | 转为低档 ($0.018/min) 收费 | 低（10-20%） | Free 成本 $12K+/月，还可承受但需监控 |
| 🔵 最好 | 继续长期免费（类似 Gemini 2.5 Flash 免费 tier 常驻） | 低（10%） | Simone 当前架构长期可持续 |

**缓解措施**：
1. **v1.3 上架前预装"内置 Key 每日 120 分钟软上限" feature flag**（§6 详述）——默认关闭，一旦 Lyria 官宣 pricing 立刻开启
2. **监测 `ai.google.dev/gemini-api/docs/models/lyria-realtime-exp` 页面变更**（建议：COO 每周扫一次，或者 Assistant 订阅 diff 通知）
3. **预算上限**：Simone 公司侧月 API 成本预算不超过 **$200/月**（老鱼个人开发者可承受上限），一旦触发预算上限立刻降级（软上限激活 / 缩短 Free 用户每日分钟数）

### 5.2 免费用户过度使用内置 Key 的成本漂移

**风险场景**：
- Free 用户发现"内置 Key 无限量"后形成口碑，Simone 变成"永久免费 AI 电台"
- Free : Paid 比例从 20:1 恶化到 50:1，总 Free 成本线性上涨

**缓解措施**：
1. **每日 120 分钟软上限**（§6）——对 Free 用户触发后温和降速 + 弹 BYOK 引导
2. **Free 用户画像监测**：v1.3 上架后每周看 App Store Analytics 的 "Session 时长分布"，重度 Free 用户（>180 min/日）占比超过 10% 就触发产品调整
3. **App Store 评论监测**：如果用户社区开始传"Simone 白嫖 AI 音乐最划算"，立刻降级软上限（从 120 min 降到 60 min）

### 5.3 Apple 抽成变化风险

**事实源**：Apple Small Business Program（2021 年推出）
- **当前状态（2026-04-19）**：Simone 作为个人开发者年收入 < $1M，享 **15% 抽成全年**
- **风险**：
  1. Simone 年收入如果破 $1M，次年续订订阅切到 **30%**（订阅第二年本来就是 15%，但 Small Business 资格没了就切 30%）
  2. Apple 取消 Small Business Program（概率极低，但政策风险存在）
  3. EU DMA / 中国反垄断压力下 Apple 被迫开放外部支付（会改变经济模型，但目前不在 Simone 可控范围）

**目前影响**：**不构成 v1.3 定价风险**。Simone 离 $1M 年收入的距离（按 50 付费用户 × $2.71/月/用户 = $1,626/年）还有 600+ 倍差距，v1.3 到 v2.0 都享 15%。这是 Simone 的隐性利好，不是风险。

### 5.4 BYOK 渗透率不及预期

**风险场景**：
- 假设的 50% BYOK 渗透率实际跌到 20-30%，临界 BYOK 96.6% 的要求彻底达不到
- 公司侧 API 成本从 $1,980 飙到 $2,534/月 以上

**缓解措施**：
1. **v1.3 上架后 30 天监测实际 BYOK 渗透**（方法：在设置页加一个埋点——实际上 App Store Analytics 不直接给这个数据，需要看 "Settings 页 BYOK 输入框交互率" 作为代理指标）
2. 如果 BYOK < 30%，立刻执行 §5.1 的 feature flag 激活 + 升级弹窗加强 BYOK 推荐

### 5.5 免费试用 → 正式订阅转化低

**草稿 §3.3 推荐给月订阅 7 天免费试用**——这带来一个风险：
- 7 天试用期用户烧 API 毛成本（中度 7 天 = 420 min × $0.06 = $25.2）但不付钱
- 如果 7 天试用 → 付费转化率 < 20%，7 天试用期的成本变成净亏
- 草稿目标是试用转化 ≥ 35%（§4.1 指标 3）——35% 是 freemium 工具类中位，但 Simone 是"即时氛围"产品，用户 7 天后是否还记得要续订是个变数

**缓解措施**：
1. 试用期第 6 天 Apple 自动发提醒邮件（StoreKit 2 原生功能），不需 Simone 额外做
2. 如果试用转化 < 20%，v1.3.1 取消月订阅试用，只留年订阅 50% off 早鸟价作为唯一优惠

## 6. 硬约束 — v1.3 必须预装"软上限"feature flag

基于 §4 临界 BYOK 96% 的结论和 §5.1 Lyria 涨价风险，**v1.3 上架版必须在代码层预装一个"内置 Key 软上限"机制**：

### 6.1 规则设计（观点推荐）

```
每日累计内置 Key 播放时长 ≥ 120 分钟 →
  触发温和降速（降低 reconnect 频率，允许自然音频中断 + 用户感知到）
  + 弹一次 bottom sheet：
    "You've been enjoying Simone for 2 hours today.
     Bring your own Gemini API key for unlimited listening,
     or upgrade to Tune for seamless experience."
    [Get API Key Guide] [Upgrade to Tune] [Maybe later]
```

### 6.2 为什么 120 min/日

- 轻度用户（15 min/日）完全不受影响
- 中度用户（60 min/日）完全不受影响
- 重度用户（180 min/日）会在每日 2 小时处感知到，**正好触发 BYOK 自我筛选**——重度用户本来就是 BYOK 核心目标人群
- 120 min × 30 天 = 3,600 min/月 × $0.06/min = **$216 单用户月上限**——Simone 公司侧可控

### 6.3 为什么是"温和降速"而不是"硬断"

- **不破坏沉浸感**（CLAUDE.md 最高产品原则）：硬断音乐正在听时停下来等于 Simone 自毁
- 温和降速 = 用户继续听，但体验逐步下降，**自然引导用户主动升级**
- 符合 Phase 2.6 "温柔陪伴"品牌人格

### 6.4 Feature flag 默认状态

- **v1.3 上架默认：关闭**（观察 v1.3 真实成本曲线再决定）
- **触发开启条件**（任一满足）：
  1. Lyria RealTime 官方开收费 + 中档或以上 pricing
  2. Simone 公司侧月 API 成本 > $150（个人开发者月可承受上限的 75%）
  3. Free 用户 BYOK 渗透率 < 5% 且 Free : Paid 比例 > 30:1

### 6.5 实施代价

- Swift 代码约 50-80 行（`AppState.dailyUsageMinutes` + `Date` 计数器 + SettingsView 显示 + bottom sheet 引导）
- 不依赖服务器（本地 `UserDefaults` 记录每日时长，次日 00:00 重置）
- 可逆：feature flag 关闭 = 零影响

**战略师推荐**：v1.3 实施 plan 里明确列这一条硬约束，由 iOS Engineer 落地为 `UsageLimiter.swift` 模块，预装但默认关闭。
