## 3. 版本号规划

### CFBundleShortVersionString（营销版本号）

```
1.2 → 1.2.1
```

**理由**：SimonePlan 定位 v1.2.1 为 v1.2 的深度化 patch，不引入新功能面、不改 UI——走 patch 位。v1.3 才是下一个 minor（商业化）。

### CFBundleVersion（build number）

**当前线上**：build `10`（Info.plist 第 18 行实测）

**v1.2.1 递增规则**：

| Build | 场景 |
|---|---|
| `11` | 预留给开发期 archive 冒烟测试（不提交 ASC） |
| `12` | TF 内测 build（小克自测 + 老鱼体感测试，对应 SimonePlan §「P4 · TestFlight 灰度 5 天」） |
| `13` | 送审 build（CEO ✅ 后提交 App Store） |
| `14+` | 审核 reject 后的修复 build（每次提交递增 1） |

**硬规则**（沿用 v1.2）：
- build 号永远只增不减，即使同一 short version
- 每次 archive 前先 `agvtool next-version -all` 或手改 Info.plist 确认递增
- TF 和 App Store 共享 build 命名空间，不能复用号码

### App Store Connect What's New 文案

由 PM 交付（参见 Release Engineer 硬规矩第 2 条「文案从 PM 拿最新版」）；Release Engineer 不起草。

## 4. 回滚方案（分三档）

> 核心原则：继承 CLAUDE.md「所有操作必须可逆」+ operating-principles.md「可逆原则」。每档都有**不对外发布**的预案（仅 TF 回滚）和**已上架**的预案（走 App Store 重新提交）。

### 小炸：单功能问题（例：某频道 Evolve 参数不合适、某维度描述词拼错）

**场景**：老鱼或外部 tester 反馈"Jazz 频道加萨克斯太频繁"这种局部问题，不是架构性崩坏。

**路径**：**热修 patch**，不回退整体版本。

```bash
# iOS repo
git checkout main
git checkout -b hotfix/v1.2.1-<具体问题>
# 改对应文件（通常是 Channel.swift 或 EvolveVocabulary.swift 某几行）
git commit -m "fix(v1.2.1): <具体问题>"
# build 递增到下一个号（例如已上架 build 13 → hotfix build 14）
# archive → TF → CEO ✅ → 提交 App Store 走常规审核
```

**耗时**：~1 天（改动 + TF 验证 + 提审）
**可逆性**：改动局限在 1-2 文件几行，revert 单 commit 即回到 v1.2.1 原始行为

### 中炸：v1.2.1 整体不对（老鱼 TF 测试 30 分钟仍疲劳 / 三维度调制器听感更差）

**场景**：SimonePlan §「验收标准」不通过——Lock 挡仍 5-10 分钟腻、10s 挡听不出加减乐器瞬间。**方向错了**，不是局部 bug。

**路径**：**回退 v1.2.1 branch，重出 v1.2.2**。

如果**还没上架**（仅 TF 阶段）：
```bash
cd "/Users/oldfisherman/Desktop/simone/simone ios"
git checkout main
git branch -D feature/v1.2.1-evolve-depth   # 或保留做研究
git reset --hard v1.2-pre-evolve-depth      # main 回到 v1.2 上架状态
git push origin main --force-with-lease     # ⚠️ 需 CEO ✅ 才能 force push
# v1.2.2 用 SimonePlan 预案的 fallback：加上频谱视觉三层耦合驱动，视听双管齐下治疲劳
```

如果**已上架**：
- **不做** `git reset --hard` 覆盖 main（已上架的 commit 要保留作为线上快照）
- 新建 branch `feature/v1.2.2-evolve-fix` 从 `v1.2-pre-evolve-depth` 切，实施 fallback 方案
- CFBundleShortVersionString: `1.2.1` → `1.2.2`
- Build 号继续递增（从当前最大值 +1，不回 10）

**耗时**：~1 周（重新实施 fallback 抓手 + TF 灰度 + 提审）
**CEO 拍板**：force push main 属于「不可逆」行为，必须回 CEO ✅（operating-principles.md 🔴 二类决策）。

### 大炸：核心音频炸（播放无声 / 崩溃率飙升 / 三维度 prompts 让 Lyria 返回损坏音频）

**场景**：App 开机没声、WebSocket 连不上、AudioEngine 崩溃——v1.2.1 触碰 PromptBuilder 导致 Lyria 协议层出问题。用户已经装了炸版本。

**路径**：**回滚到 v1.2 的 `v1.1.1-pre-v1.2-merge` tag**（等等——读 tag 字面意义容易误解，这个 tag 指的是 "v1.1.1 时代、在 v1.2 merge 发生之前"的状态，即 **v1.1.1 最后稳态**，不是 v1.2 稳态）。

**更稳的大炸回滚目标**：应该回到 **`dec17ab`**（v1.2 build 10 线上版），**不是** `v1.1.1-pre-v1.2-merge`。后者是 v1.1.1 稳态、没有 Fog 视觉，会让已经装 v1.2 的用户体验倒退一个大版本。

**修正后的大炸预案**：

```bash
cd "/Users/oldfisherman/Desktop/simone/simone ios"
# 新建紧急 branch 指向 v1.2 线上 tag
git checkout -b hotfix/emergency-rollback-to-v1.2 v1.2-pre-evolve-depth
# CFBundleShortVersionString 改为 1.2.2（不能复用 1.2 / 1.2.1）
# CFBundleVersion 递增（例如已发到 13，改为 14）
# 不做任何代码改动——就是 v1.2 build 10 的内容重新打包
# archive → 跳过 TF 外测 → 直接 submit with expedited review request
```

**真正回到 `v1.1.1-pre-v1.2-merge` 的场景**：只有 v1.2 本身也被发现有隐藏大炸（两版接连出事）才用。单独 v1.2.1 炸不需要退两版。

**耗时**：archive 1 小时 + Apple expedited review 通常 24-48 小时
**CEO 拍板**：提 expedited review 属于「对外发布」，🔴 必须 CEO ✅
**附加动作**：v1.2.1 分析 postmortem、通知已升级的 TF 用户

### 回滚决策矩阵

| 征兆 | 档位 | 动作 | CEO 拍板？ |
|---|---|---|---|
| 1-2 频道体感偏差 | 小炸 | 热修 patch 1.2.1 → 1.2.1 build +1 | 送审时 ✅ |
| Evolve 整体方向错 | 中炸 | 1.2.2 重出，可能加 fallback | 方向变更 + 送审都要 ✅ |
| 播放炸 / 崩溃飙升 | 大炸 | 紧急 1.2.2 = v1.2 build 10 内容 + expedited | expedited 必 ✅ |

## 5. 双 repo commit/push 顺序

**建议：先 push iOS repo，后 push 主 repo。**

**一句理由**：iOS repo 是用户真的会装到手机上的东西——上架出问题是炸；主 repo 是内部文档，晚几分钟同步无损。先把"真东西"的状态固化了，再用文档追述它。

**完整流水（v1.2.1 ship 日顺序）**：

1. iOS repo：feature branch merge 回 main（`--no-ff`）
2. iOS repo：打 tag `v1.2.1-shipped`
3. iOS repo：`git push origin main --tags`
4. iOS repo：Xcode archive → ASC 上传 build 13 → submit（CEO ✅ 后）
5. 【等审核通过通知】
6. **主 repo**：更新 `SimonePlan.md` v1.2.1 状态 📋 → ✅、更新 `CLAUDE.md` v1.x 状态表、写当日 `docs/daily/YYYY-MM-DD.md`
7. 主 repo：`git commit` + `git push origin <branch>`

**例外**：如果主 repo 的改动是 v1.2.1**立项前置**（本预案这种），就正常先 push 主 repo——此时 iOS repo 没改动，谈不上顺序。顺序规则只在**同一次 ship 动作**里生效。
