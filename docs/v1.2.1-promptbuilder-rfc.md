# v1.2.1 PromptBuilder 三维度调制 RFC

> **作者**：iOS Engineer（临时代理 · 替 COO 小克出 RFC）
> **日期**：2026-04-19
> **主题**：Evolve 深度化 · `instruments` / `density` / `energy` 三维度调制替代 temperature 抖动
> **上游规划**：`SimonePlan.md` §「v1.2.1 —— Evolve 深度化」+ `docs/v1.2.1-release-plan.md`
> **状态**：📋 待 COO / CEO review → 通过后由 iOS Engineer 正式执行
> **本 RFC 不含任何代码改动**（只出方案 + 伪码示例），CEO review 要看的是「**方向**」和「**可逆边界**」，不是实现细节

---

## 0. TL;DR（给 CEO 用的 30 秒版）

**现状**：v1.1.1 的 Evolve 是「`EvolveVocabulary` 随机挑 1 件乐器 `±`、1 个 energy 词、70% 概率 1 个 texture 词」拼到 prompt 尾巴 + `temperature ± 0.1` / `guidance ± 0.25` / `density ± 0.08` 抖动。**问题**：拼到 prompt 尾巴的词汇是**无状态**的——每次 Evolve 都从空白重掷一次骰子，上一次加了萨克斯下一次可能再加一遍萨克斯；能量词也没方向性，只是随机词。**听感**：无叙事的玄学波动，Lock 5-10 分钟腻、Evolve 开 15 分钟腻。

**新方案**：把 Evolve 从「无状态词汇重掷」升级为「**有状态的三维度连续调制**」：
- **乐器池**（instruments）：每频道定义 `core / accent / optional` 三层池，维护**当前在位的 accent/optional 集合**，每次 tick `±1` 件（有记忆的加减，不重复掷骰）
- **密度**（density）：`0.3-1.0` 标量，`±0.15` 连续游走（带边界反射），映射到 `sparse / moderate / dense / full arrangement` 四档描述词
- **能量**（energy）：`0.3-1.0` 标量，同上，映射到 `laid-back / steady / driving / intense` 四档描述词
- **每 tick 只挑 1-2 个维度动**，不全改，避免突变
- **Lock 挡**：三个维度**完全冻结当前状态**，不抖 temperature——真正的「稳住别变」

**可逆性**：`git revert` 三个 commit 回到 v1.2 行为。`PromptBuilder` 对外签名不变，`LyriaClient` / `AudioEngine` / `Views` 零改动。

**关键决策点（CEO 看这里）**：
1. **三维度是否暴露到 UI？** 本 RFC 方案 = `false`（后台状态，v2.0 再开放给 Studio 档）——符合 SimonePlan §红线。
2. **Lock 挡是否真冻结 temperature？** 本 RFC 方案 = `true`（不抖了）——这是"撑 30 分钟"的单项提升最大的改动，也是最容易被用户察觉的行为变化（Lock 听感会变更"定"）。
3. **评估标准**：老鱼 TF 自测 30+ 分钟不想切即通过；否则触发 SimonePlan §预案——补做 v2.0 抓手 A 的频谱视觉耦合（视听双管齐下治疲劳）。

## 1. 当前 PromptBuilder 实现摘要 + 根因分析

### 1.1 当前代码路径（2026-04-19 实测）

**`PromptBuilder.swift` 全文（22 行）**：
- `build(style:)` → 单个 WeightedPrompt，`text = style.prompt`，`weight = style.promptWeight`
- `evolveVariant(style:)` → 调 `EvolveVocabulary.variant(for: category)` 拼接到 `style.prompt` 尾巴，返回单个 WeightedPrompt

**`EvolveVocabulary.swift:11-47`**：按 `StyleCategory` switch 返回 6-8 个乐器词。
**`EvolveVocabulary.swift:52-61`**：`energy`（7 词）+ `texture`（6 词）两个**跨频道共用**的词池。
**`EvolveVocabulary.swift:67-77`** `variant(for:)`：
```swift
let addRemove = Bool.random() ? "+" : "-"    // 纯随机
let energyWord = Self.energy.randomElement() ?? "softer"
// 70% 概率拼 texture
return " ± <instrument> + <energy_word> [+ <texture>]"
```

**`AppState.swift:486-508` `evolve()`**：
1. `PromptBuilder.evolveVariant(style:)` → `sendPrompts(prompts)`
2. `temperature += Float.random(in: -0.1...0.1)`（裁剪到 0.1-3.0）
3. `guidance += Float.random(in: -0.25...0.25)`（裁剪到 0.0-6.0）
4. 若 `density >= 0`：`density += Float.random(in: -0.08...0.08)`（裁剪到 0-1）
5. 若 `brightness >= 0`：同 density
6. `sendConfig(buildFullConfig())`

**触发频率**：`EvolveMode` 现有四档 `.locked / .auto30s / .auto1m / .auto5m`（注意：SimonePlan 文档中写的 "10s 挡" **已不存在**——v1.1.1 AppState 注释写 `30s is the fastest tier — 10s felt twitchy on Lyria's settle time`）。默认 `.locked`。

### 1.2 根因分析：为什么 Lock 挡 5-10 分钟腻、Evolve 开 15 分钟腻

| 现象 | 根因 | 本 RFC 对治 |
|---|---|---|
| **Lock 挡 5-10 分钟腻** | Lock 模式确实 return early（`AppState.swift:462`），但 `temperature=1.1` 本身是高抖动值（Lyria 默认），Lyria 内部每个 audio chunk 都在高随机度下重新解码——即使 prompt 不变，音乐也在"内部熵涌动"。用户期望 Lock=完全稳住，实际是"不再外部施压，但内部还在乱" | Lock 挡下 `temperature` 降到 `0.7`（或者更狠 `0.5`）并**冻结**，让 Lyria 内部也趋稳 |
| **Evolve 开 15 分钟腻** 之 A | 词汇重掷无记忆：第 10 分钟加了"saxophone"，第 11 分钟 50% 概率就被 `-` 掉——用户还没"认出"这件乐器，它就没了。乐器感知需要时间 | 维护 `activeAccents: Set<String>` / `activeOptionals: Set<String>` 状态，真加真减，不重复掷 |
| **Evolve 开 15 分钟腻** 之 B | energy 词池（`denser / minimal / lush / sparse / warmer / brighter / softer`）是**互相矛盾**的——第 10 分钟加"denser"、第 11 分钟加"sparse"、Lyria 只看到 prompt 末尾一串自相矛盾的形容词，权重下降，**变成噪音** | 换成有**单调方向**的标量映射（`density ∈ [0.3, 1.0]`），每次只小步游走，描述词从四档里**选一个**而不是"加一个" |
| **Evolve 开 15 分钟腻** 之 C | prompt 只无限拼接，没有 `weightedPrompts` 多路混合。Lyria API 本支持多 WeightedPrompt（reference_lyria_realtime §WeightedPrompt），但我们只用了单路 | v1.2.1 先不动多路——**多路是 v2.0 的事**（Studio 档多风格混合）。v1.2.1 仍保持单 prompt，但 prompt 文本由三维度**重新组装**（不是 append），长度不增长 |
| **Evolve 开 15 分钟腻** 之 D | `temperature / guidance` 同时抖 + `density / brightness` 同时抖——4 个维度全在动，用户听到的是白噪声式漂移而不是"有方向的演变" | 每次 tick 只改 1-2 个维度。`temperature / guidance` 在新方案里**不再由 Evolve 抖**（交给 Studio 档手动控） |

### 1.3 新方案的"叙事性"来源

用户听觉能感知的变化粒度大约是 **20-40 秒**（一段 verse/chorus 的长度）。当前词汇轮转的时间尺度也是 30s-5min，但每次变化都是"随机重掷"——没有**连续性**。

新方案的核心是：三维度连续游走 = **每次变化都以上一次为起点**，形成可识别的音乐曲线（渐进加乐器 → 稳住 → 减乐器；能量上升 → 平台期 → 回落）。这对齐 Phase 2.6 §Evolve 原则「加减乐器、密度变化、轻微能量变化」。

## 2. 三维度调制方案

### 2.1 维度 A：`instruments`（乐器池调制 · 有状态加减）

**数据结构**（新增在 `Channel.swift` 或 `StyleCategory.swift`，二选一由实施时决定）：

```swift
// 伪码（示意结构，实施时 Channel.swift 和 StyleCategory.swift 二选一承载）
struct InstrumentPool {
    let core: [String]     // 永不移除，奠定频道身份（3-4 件）
    let accent: [String]   // 1m 挡粒度 ±1，给频道"点色彩"（3-5 件）
    let optional: [String] // 30s 挡粒度 ±1，锦上添花（3-5 件）
}

extension StyleCategory {
    var instrumentPool: InstrumentPool {
        switch self {
        case .lofi:
            return InstrumentPool(
                core:     ["soft piano", "mellow beats", "vinyl warmth"],
                accent:   ["rhodes", "warm pads", "lazy guitar", "lofi bass"],
                optional: ["subtle rain", "tape hiss", "vocal chops", "muted trumpet"]

        case .jazz:
            return InstrumentPool(
                core:     ["walking bass", "brushed drums", "piano trio"],
                accent:   ["tenor sax", "muted trumpet", "vibraphone", "hammond organ"],
                optional: ["flute", "clarinet", "soft strings", "gentle shaker"]

        case .rnb:

                optional: ["soft horns", "vocal pad", "808 sub", "chimes"]

        }

```

**状态持有**（在 `AppState` 新增）：
```swift
// 伪码
private var activeAccents: Set<String> = []     // 当前在位的 accent 件
private var activeOptionals: Set<String> = []   // 当前在位的 optional 件
```

初始化时：selectStyle 时根据 `style.category` 填充一个默认初始集合（例如：accent 池前 2 件 + optional 池前 1 件）。

**Evolve tick 对 instruments 的操作**：
```swift
// 伪码（每次 tick 按挡位决定影响哪层）
func evolveInstruments(category: StyleCategory, mode: EvolveMode) {
    let pool = category.instrumentPool
    switch mode {
    case .auto30s:
        // 只动 optional，±1 件
        toggleOne(from: pool.optional, in: &activeOptionals, maxSize: 2)
    case .auto1m:
        // 主动 accent，偶尔动 optional
        if Bool.random(0.7) {
            toggleOne(from: pool.accent, in: &activeAccents, maxSize: 3)
        } else {
            toggleOne(from: pool.optional, in: &activeOptionals, maxSize: 2)
        }
    case .auto5m:
        // 整组 reshuffle 一次：accent 随机抽 2 件、optional 随机抽 1 件
        activeAccents = Set(pool.accent.shuffled().prefix(2))
        activeOptionals = Set(pool.optional.shuffled().prefix(1))
    case .locked:
        return
    }
}

func toggleOne(from pool: [String], in active: inout Set<String>, maxSize: Int) {
    // 加/减决策：如果 active.count >= maxSize 强制减、<= 0 强制加、否则各 50%
    let shouldAdd: Bool
    if active.count >= maxSize { shouldAdd = false }
    else if active.isEmpty { shouldAdd = true }
    else { shouldAdd = Bool.random() }

    if shouldAdd {
        let candidates = pool.filter { !active.contains($0) }
        guard let pick = candidates.randomElement() else { return }
        active.insert(pick)
    } else {
        guard let drop = active.randomElement() else { return }
        active.remove(drop)
    }
}
```

### 2.2 维度 B：`density`（密度 · 连续游走 → 四档描述词）

**状态**：
```swift
// AppState 新增
var currentDensity: Float = 0.6   // 默认中等
```

**游走规则**：
```swift
// 每次 tick（如果选中 density 维度）
currentDensity = max(0.3, min(1.0, currentDensity + Float.random(in: -0.15...0.15)))
```

**映射到 prompt 描述词**：
```swift
func densityPhrase(_ v: Float) -> String {
    switch v {
    case ..<0.45: return "sparse arrangement"
    case ..<0.65: return "moderate arrangement"
    case ..<0.85: return "dense arrangement"
    default:      return "full arrangement"
    }
}
```

**Lyria API 层面**：同时也把 `currentDensity` 写进 `musicGenerationConfig.density`（float 字段本就支持）。这样描述词和数值双保险——描述词塑形、数值微调。

### 2.3 维度 C：`energy`（能量 · 连续游走 → 四档描述词）

**状态**：
```swift
// AppState 新增
var currentEnergy: Float = 0.55   // 默认偏静
```

**游走规则**（和 density 同）：
```swift
currentEnergy = max(0.3, min(1.0, currentEnergy + Float.random(in: -0.15...0.15)))
```

**映射**：
```swift
func energyPhrase(_ v: Float) -> String {
    switch v {
    case ..<0.45: return "laid-back feel"
    case ..<0.65: return "steady groove"
    case ..<0.85: return "driving rhythm"
    default:      return "intense energy"
    }
}
```

**Lyria API 层面**：`energy` **没有** 对应的 Lyria config 字段（reference_lyria_realtime §LiveMusicGenerationConfig 只有 density/brightness/bpm/temperature/guidance/top_k）。所以 energy 只走 prompt 描述词这一路。

### 2.4 三维度独立触发（避免突变）

```swift
// 伪码（在 AppState.evolve() 替换现版）
private func evolve() {
    guard let style = selectedStyle,
          lyriaClient.connectionState == .connected else { return }

    // 1. Lock 挡完全冻结（含 temperature）
    guard evolveMode != .locked else { return }

    // 2. 随机挑 1-2 个维度这次要动
    let allDims: [EvolveDimension] = [.instruments, .density, .energy]
    let picked = allDims.shuffled().prefix(Int.random(in: 1...2))

    // 3. 分别调制
    for dim in picked {
        switch dim {
        case .instruments: evolveInstruments(category: style.category, mode: evolveMode)
        case .density:     currentDensity = walk(currentDensity)
        case .energy:      currentEnergy = walk(currentEnergy)
        }
    }

    // 4. 重建 prompt 文本（不 append！）
    let prompt = PromptBuilder.build(
        style: style,
        activeAccents: activeAccents,
        activeOptionals: activeOptionals,
        density: currentDensity,
        energy: currentEnergy
    )
    lyriaClient.sendPrompts(prompt)

    // 5. 配置层：density 走 Lyria 原生字段，不再抖 temperature/guidance/brightness
    var config: [String: Any] = ["density": currentDensity]
    lyriaClient.sendConfig(config)
}
```

### 2.5 重构后 `PromptBuilder` 伪码

```swift
// 伪码 — 新 PromptBuilder.swift 结构
enum PromptBuilder {
    static func build(style: MoodStyle) -> [WeightedPrompt] {
        // 非 evolve 模式（selectStyle / regenerate）仍走纯 style.prompt
        [WeightedPrompt(text: style.prompt, weight: style.promptWeight)]
    }

    /// v1.2.1 · 带三维度调制的 evolve prompt 组装
    static func build(
        style: MoodStyle,
        activeAccents: Set<String>,
        activeOptionals: Set<String>,
        density: Float,
        energy: Float
    ) -> [WeightedPrompt] {
        var parts: [String] = [style.prompt]

        // instruments 部分：active 件列表（with natural language glue）
        let instrs = (Array(activeAccents) + Array(activeOptionals)).sorted()
        if !instrs.isEmpty {
            parts.append("with " + instrs.joined(separator: ", "))
        }

        // density + energy 描述词
        parts.append(densityPhrase(density))
        parts.append(energyPhrase(energy))

        let text = parts.joined(separator: ", ")
        return [WeightedPrompt(text: text, weight: style.promptWeight)]
    }

    // densityPhrase / energyPhrase 私有辅助
}
```

**关键性质**：
- `build(style:)` 单参版本签名不变——`AppState.applySelection()` / `regenerate()` / `sendCurrentPrompts()` 三处调用点零改动
- 新的 `build(style:activeAccents:activeOptionals:density:energy:)` 只在 `evolve()` 里调
- `EvolveVocabulary.swift` 的词典仍保留一个版本（Phase §可逆性），但 `variant(for:)` 方法不再被调用——双向可逆到 v1.2

## 3. 量化"撑 30 分钟不疲劳"的测试方案

### 3.1 主观 metric（老鱼自测 · TF 阶段）

| 挡位 | 时长 | 期望体感 | 量化 |
|---|---|---|---|
| **Lock** | 30 min | "几乎没变，但我能专注" | 主观打分 1-5，目标 ≥ 4；不能切歌念头出现 ≥ 20 min 算通过 |
| **30s** | 30 min | "每隔一会儿能听出有乐器加减，但没断档感" | 听到 5+ 次"有变化"的瞬间 = 通过 |
| **1m** | 60 min | "明显的能量曲线——起、稳、落" | 至少 1 次感知到"曲线转折" = 通过 |
| **5m** | 90 min | "像 DJ 慢慢换段子" | 至少 1 次感知到"整段换气氛" = 通过 |

**硬门槛**：**老鱼 TF 自测 30+ 分钟不想切**（SimonePlan §验收）。不通过 → 触发 fallback（v2.0 抓手 A 的频谱视觉耦合前置）。

### 3.2 客观 metric（日志侧）

在 `evolve()` 里加**调试 log**（`#if DEBUG` gate，Release 不发）：

- 对照老鱼自述体感的时间戳，核对是否有对得上的调制事件

### 3.3 A/B 对照（可选 · 如果 P4 灰度不过）

保留一个**开关 flag**（`UserDefaults.bool(forKey: "useLegacyEvolve")` 默认 false），打开后回退到 v1.2 的 `EvolveVocabulary.variant` + temperature 抖动。老鱼可在"设置 → Advanced"里切换对比。

**注**：这个开关只在**内部 debug build** 暴露；Release build 代码里保留但 UI 不暴露。通过老鱼 Xcode Console 里打字或者 hidden gesture 触发即可。

### 3.4 不做的测试

- **自动化 "fatigue detection"**：AI 无法客观判断"腻不腻"，老鱼自测就是真理
- **众包 A/B**：v1.2.1 是付费前置内测，样本太小
- **频谱熵分析**：无法解释听感

## 4. 技术风险 + 三档回滚点

### 4.1 风险清单

| 风险 | 触发场景 | 缓解 | 严重度 |
|---|---|---|---|
| **R1 · Lyria prompt 长度超限** | active 件太多 + 两个描述词，prompt 可能超过几百字符 | active 上限硬编码 accent ≤ 3 + optional ≤ 2 = 最多 5 件；加 style.prompt 约 30 词 + 两个描述词约 10 词 ≈ 60 词，Lyria 实测支持 | 低 |
| **R2 · 乐器池 ≠ style.prompt 已有乐器 → 冲突** | 例如 Lo-fi `style.prompt` 已含 "rhodes"，active accent 又加了 "rhodes" → "with rhodes, rhodes" | 组装时去重（`Set`）；最坏冲突也只是让 Lyria 对某件乐器权重更高，不会崩 | 低 |
| **R3 · Lyria 对 density config 字段的响应不稳定** | reference_lyria_realtime §LiveMusicGenerationConfig 说 density 是 0-1 默认 —，但实测可能和 prompt 描述词冲突 | 初期只靠 prompt 描述词塑形，config.density **与描述词保持同向**；如果起反作用，拆掉 config.density 只留描述词 | 中 |
| **R4 · Lock 挡降 temperature 导致 Lyria 过于单调** | temperature 从 1.1 → 0.7 可能让 Lyria 循环感过强 | Lock 挡降到 0.8（比现在的 1.1 稳，不至于跟"采样"一样）；实测不够再调；**最坏情况 Lock 挡不改 temperature**（只是不抖 ±0.1，停在 1.1） | 中 |
| **R5 · active state 没持久化，App 冷启动丢失** | 用户听到第 20 分钟 kill App 再开，所有 accent/optional 归零 | 不持久化——这个设计决策是**特征不是 bug**，重开 App 等于电台换台，反而符合"新鲜感" | — |
| **R6 · currentDensity / currentEnergy 命名冲突 AppState 已有 `density / brightness / mood / energy` 字段** | AppState 已有 `energy: Float = 0.5` 和 `density: Float = -1`。新字段加 `currentDensity` / `currentEnergy` 避免名字冲撞，但语义上仍需和老字段协调 | 老 `energy` / `density` 是 "v1.0 Max 档未暴露的手动 config 占位"，v1.2.1 里**不使用**，保留字段避免破坏 Codable。新逻辑只读写 `currentDensity` / `currentEnergy` | 低 |
| **R7 · HMR 不触发 evolveTimer 重建** | SwiftUI HMR 改 AppState 字段定义后老 timer 还在跑旧闭包 | 老鱼自测纪律：音频/架构改动后硬重启模拟器（CLAUDE.md 已规定） | 低（流程管控） |
| **R8 · 灰度发现"撑 30 分钟"不达标** | 三维度听感和 v1.2 差距不够大 | 触发 release-plan §中炸预案：v1.2.2 加频谱三层耦合（v2.0 抓手 A 部分前置） | 中 |

### 4.2 三档回滚点（对齐 release-plan §4 小炸/中炸/大炸）

| 档 | 场景 | 动作 | 代码/发布层面 |
|---|---|---|---|

### 4.3 可逆性保证

- **`EvolveVocabulary.swift` 不删**：里面的 `variant(for:)` 方法保留，只是 `AppState.evolve()` 不再调用。如果回退，改一行就能切回 v1.2 行为
- **`PromptBuilder.build(style:)` 签名不变**：`applySelection` / `regenerate` / `sendCurrentPrompts` 三处调用点零改动——这是**对外契约稳定**
- **UserDefaults schema 零改动**：三维度状态是**运行时内存状态**，不写盘。即使 v1.2.1 炸了，用户没有"坏的持久化数据"需要清理
- **Lyria 协议层零改动**：不引入新字段、不改 camelCase 转换表——`LyriaClient.sendConfig` / `sendPrompts` 都按现契约用

## 附录 A：为什么不用多 WeightedPrompt（v1.2.1 暂不做）

Lyria API 原生支持多 WeightedPrompt：`[{text: "jazz trio", weight: 0.7}, {text: "with saxophone", weight: 0.3}]`。理论上比拼接字符串更优雅（权重独立调）。

**本版不做的理由**：
1. v1.2.1 只有 1 周窗口，多 prompt 的权重调校需要额外灰度轮次
2. 多 prompt 是 SimonePlan §v2.2+ 长尾的"多风格混合"功能 stake，不该提前消耗
3. 单 prompt 重组字符串**足以**表达三维度变化——prompt engineering 上限高于协议能力

**v2.0 再做**：Studio 档多风格混合 = 多 WeightedPrompt 的自然用法。

## 附录 B：active 初始化策略

`selectStyle(style)` 时重置 active 集合：
```swift
// 伪码
private func resetActiveSets(for category: StyleCategory) {
    let pool = category.instrumentPool
    // 默认初始：accent 前 2 件、optional 前 1 件
    activeAccents = Set(pool.accent.prefix(2))
    activeOptionals = Set(pool.optional.prefix(1))
    // density / energy 回默认中值
    currentDensity = 0.6
    currentEnergy = 0.55
}
```

这保证了切台时有新的基线 + `.locked` 挡下仍有 active 乐器被拼进 prompt（即使不再演变）。
